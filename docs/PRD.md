# CS2 Demo AI Coach PRD

## 1. 产品概述

### 产品名称

CS2 Demo AI Coach

### 一句话定位

用户上传 CS2 demo 后，系统自动分析个人习惯、团队风格、关键失误，并为五人队制定专属战术和训练建议。

### 产品目标

把一场 CS2 demo 转换成可执行的复盘结论：

- 你有哪些坏习惯
- 你适合打什么位置和角色
- 哪些回合本可以处理得更好
- 五个人整体有什么风格、强项和短板
- 根据五个人的特点制定专属战术

第一阶段重点不是做完整 demo 播放器，而是做“证据驱动的 AI 复盘报告”。

## 2. 目标用户

### 核心用户

- CS2 五排车队
- 半职业、校园、业余战队
- 想提升但不会复盘的普通玩家
- 战队教练或队长

### 用户痛点

- demo 很长，人工复盘费时间
- 普通玩家不知道自己具体哪里打得不好
- 队伍缺少明确角色分工
- 战术经常照抄职业队，但不适合自己的五个人
- 复盘容易变成甩锅，缺少客观数据证据
- 队伍知道“打得乱”，但不知道应该怎么改

## 3. 核心价值

上传一场 demo 后，系统输出：

- 每个人的坏习惯
- 每个人适合的位置和角色
- 每个关键回合应该怎么打更好
- 五人整体风格分析
- 队伍短板和强项
- 基于五个人习惯定制的战术
- 下一阶段训练清单

产品成功的核心标准：

> 用户看到报告后觉得“这确实是在分析我这场比赛，而不是通用废话”。

## 4. MVP 范围

### MVP 做什么

- 支持上传 `.dem`
- 解析一场完整比赛
- 自动识别地图、比分、玩家、回合、阵营
- 用户选择己方五名玩家
- 用户选择重点分析对象
- 输出个人复盘报告
- 输出团队复盘报告
- 输出 2-3 套专属战术
- 标记 5-10 个关键回合
- 每条建议绑定证据：回合、时间、位置、事件

### MVP 地图范围

第一版建议只支持 1 张地图，优先：

- Mirage

后续扩展：

- Inferno
- Dust2
- Ancient
- Nuke
- Anubis
- Vertigo

### MVP 暂不做

- 完整 2.5D demo 播放器
- 实时观战分析
- 语音沟通分析
- 职业级战术库
- 自动判断玩家真实意图
- 准星控制和鼠标细节分析
- 反作弊类判断

## 5. 核心用户流程

1. 用户上传 demo。
2. 系统解析 demo。
3. 用户选择自己的队伍和自己的 ID。
4. 系统生成比赛概览。
5. 用户查看个人分析。
6. 用户查看团队分析。
7. 用户查看关键回合复盘。
8. 用户查看专属战术和训练计划。
9. 用户导出报告或分享链接。

## 6. 功能模块

### 6.1 Demo 上传与解析

#### 需求

- 支持上传 `.dem`
- 显示上传进度
- 显示解析进度
- 解析失败时给出明确原因
- 自动识别地图、比分、队伍、玩家、回合数
- 解析击杀、死亡、伤害、道具、经济、C4、位置轨迹

#### 解析输出数据

- 比赛基础信息
- 回合列表
- 玩家列表
- 每回合关键事件
- 每个玩家的统计指标
- 每个玩家的位置和路径摘要
- 每个队伍的经济变化
- 投掷物事件
- C4 事件

### 6.2 比赛概览

#### 展示内容

- 地图
- 比分
- 阵营胜率
- 每回合胜负
- 最关键的 3 个转折回合
- 队伍总体评价
- 本场最大问题
- 本场最大优势

#### 示例输出

```text
你们本场最大问题是 T 方中期掉人过多。第 5、8、11、15 回合都出现了无补枪首死，导致默认控图失败。
```

### 6.3 个人坏习惯分析

#### 输出内容

每个玩家输出：

- 3-5 个坏习惯
- 每个坏习惯的严重程度
- 证据回合
- 具体改法
- 训练建议

#### 坏习惯类型

- 单走首死
- 无补枪距离
- 重复 peek
- 道具无收益
- 闪到队友
- 下包后站位过激
- 优势局继续找人
- 残局过早暴露位置
- 回防过慢
- 经济决策不统一
- 固定路线被针对

#### 示例输出

```text
坏习惯：你经常在默认控图阶段独自前压。

证据：
第 7 回合 0:24，你在 top mid 首死，最近队友距离 1300 units，死亡后 5 秒内无人补枪。
第 11 回合 0:31，你在 underpass 单走被击杀，队伍失去中路控制。

建议：
你更适合等队友 flash 或 connector 补位后再争夺中路，不要在默认阶段独自扩大接触面。
```

### 6.4 适合位置与角色推荐

#### 角色类型

- Entry
- Second entry
- Support
- Lurker
- AWPer
- Anchor
- Rotator
- Clutcher
- IGL 倾向

#### 判断指标

- opening duel win rate
- first death rate
- trade kill rate
- traded death rate
- utility effectiveness
- clutch win rate
- rotate timing
- site hold success
- post-plant survival
- lurk impact

#### 示例输出

```text
你更适合打 second entry / support。

原因：
你的补枪成功率高于队伍平均值，但首杀对枪胜率偏低。你不适合作为第一个进点的人，更适合跟在 entry 后面完成补枪，同时负责关键闪光。
```

### 6.5 关键回合复盘

#### 自动挑选的关键回合类型

- 经济转折回合
- 5v4 被翻盘
- 4v2 被翻盘
- 首死导致崩盘
- 下包后处理失败
- 残局失败
- 道具失误
- 执行时间过晚
- 防守重心判断错误

#### 每个回合输出

- 回合结果
- 关键时间线
- 主要失误
- 更好的打法
- 相关玩家
- 可选小地图轨迹

#### 示例输出

```text
第 12 回合：4v3 优势局被翻盘

问题：
你们在拿到人数优势后继续分散找人，没有收缩到包点建立交叉火力。

更好选择：
A 点下包后，entry 不应该继续压 CT，而是回到 triple 附近和 palace 队友形成交叉。jungle 烟消失前需要一名队友主动补烟或架住 connector。
```

### 6.6 团队风格分析

#### 输出内容

- 队伍偏快攻还是慢控
- T 方是否依赖单人突破
- CT 方是否容易过早转点
- 谁是稳定输出点
- 谁容易成为突破口
- 哪个区域最容易掉人
- 哪种局势最容易输
- 道具协同是否足够
- 经济决策是否统一

#### 示例输出

```text
你们是一支偏快节奏队伍，但默认控图质量不足。T 方前 40 秒经常出现单点接触，导致战术还没展开就少人。
```

### 6.7 专属战术制定

#### 战术生成原则

战术必须根据五个人的实际习惯生成，而不是套模板。

每套战术包含：

- 适用地图
- 适用阵营
- 适用经济条件
- 战术目标
- 五人分工
- 开局站位
- 道具分配
- 执行时机
- 失败后的应急方案
- 适合这支队伍的原因

#### MVP 输出

- 2 套 T 方战术
- 1 套 CT 方默认防守
- 1 套 eco/半起策略，后续加入

#### 示例输出

```text
战术：Mirage T 方中路控图转 A 夹击

适合原因：
你们队伍中 PlayerA 补枪能力强，PlayerB 道具收益高，但 PlayerC 单摸死亡率高。因此不建议让 PlayerC 独自 lurk B，而是让他负责 palace 后点牵制。

分工：
PlayerA：second entry，从 connector 夹 A
PlayerB：support，负责 window smoke、connector smoke
PlayerC：palace 牵制
PlayerD：entry，从 ramp 进点
PlayerE：断后，防止 B 小前压
```

### 6.8 训练计划

#### 输出内容

根据坏习惯和团队短板生成训练任务。

#### 示例输出

```text
本周训练重点：

1. 默认控图补枪距离
目标：两人距离保持在 600 units 内，减少无交易首死。

2. 下包后站位
目标：练习 A 点 3 套 post-plant 交叉火力。

3. 道具 timing
目标：T 方执行前 8 秒完成关键烟闪，不再 20 秒以下才开始爆弹。
```

## 7. 分析指标体系

### 个人指标

- K/D
- ADR
- KAST
- opening duel win rate
- first death rate
- first kill rate
- trade kill rate
- traded death rate
- time to trade
- clutch win rate
- utility damage
- flash assists
- enemies flashed
- teammates flashed
- survival rate after advantage
- death position repeat rate

### 团队指标

- T 方默认控图成功率
- CT 方防守站位稳定性
- 回防平均时间
- 下包后胜率
- 5v4 胜率
- 4v3 胜率
- eco 局胜率
- force buy 成功率
- 道具协同效率
- 经济同步程度
- 区域失守频率

### 位置和角色判断

- Entry：看首杀尝试、首杀成功率、第一接触质量
- Second entry：看补枪成功率、跟进距离、进点存活率
- Support：看道具收益、闪光助攻、执行配合
- Lurker：看单摸收益、断后成功率、信息价值
- Anchor：看包点防守成功率、拖延时间、多杀能力
- Rotator：看回防速度、转点准确率、残局参与度
- Clutcher：看残局胜率、残局伤害、时间处理

## 8. AI 分析原则

### 基本原则

AI 不直接凭感觉评价玩家。系统先通过规则引擎和统计指标找出证据，再让 AI 解释和生成建议。

### 必须遵守

- 每条负面建议必须绑定具体证据
- 每条战术建议必须解释为什么适合这五个人
- 不输出无法从 demo 推断的结论
- 不根据单个回合过度判断长期习惯
- 对低置信度结论要标注“可能”

### 禁止输出

- “你意识不好”这类无证据评价
- “你枪法不行”这类无法从 demo 准确判断的结论
- 没有回合、时间、事件支撑的泛泛建议
- 没有分工细节的空泛战术

## 9. 技术方案

### 推荐架构

```text
前端
React / Next.js

后端
Go / Node.js

Demo Parser
demoinfocs-golang

数据处理
PostgreSQL + Redis

文件存储
S3 / R2 / 本地存储

AI 分析
DeepSeek V4 Pro / 其他大模型

可视化
Canvas / PixiJS
```

### 分析流程

```text
上传 demo
→ parser 解析原始事件
→ 生成结构化比赛数据
→ 规则引擎计算指标
→ 筛选关键问题和证据
→ 压缩成 AI 输入
→ AI 生成报告和战术
→ 前端展示
```

### AI 输入策略

不要把逐 tick 原始数据直接发给模型。

推荐输入：

- 比赛摘要
- 回合摘要
- 关键事件
- 玩家指标
- 问题证据
- 位置路径关键点
- 规则引擎初筛结果

目标控制：

- 常规单场分析：30k-80k input token
- 深度复盘加战术：80k-150k input token
- 输出：8k-40k token
- 单场 AI 成本目标：0.3-1 元以内

## 13. 最大风险与解决方案

### 风险 1：demo 解析不稳定

解决方案：

- 先支持固定地图和固定 demo 来源
- 解析失败时保留错误日志
- 建立 demo 样本库做回归测试

### 风险 2：AI 胡说

解决方案：

- 所有建议必须绑定具体证据
- 规则引擎先筛问题
- AI 只做解释、排序和表达
- 对低置信度结论做标注

### 风险 3：成本过高

解决方案：

- 不喂逐 tick 原始数据
- 压缩成回合摘要和关键证据
- 常见战术模板缓存
- AI 只做定制化修改

### 风险 4：用户觉得不准

解决方案：

- 每条建议增加“有用/不准”反馈
- 报告展示证据链
- 不做过度推断
- 允许用户选择重点分析对象和目标位置

### 风险 5：战术太空泛

解决方案：

- 战术必须绑定五人角色
- 战术必须说明适合原因
- 战术必须包含开局站位、道具、执行时机、失败预案

## 14. MVP 成功标准

第一版只要做到以下几点就算成功：

- 能指出具体回合
- 能指出具体死亡、道具、经济问题
- 能说出每个人的习惯差异
- 能给出适合这五个人的战术分工
- 用户愿意上传下一场 demo 继续分析

## 15. 推荐第一阶段执行方案

第一阶段不要做复杂 UI 和 2.5D 回放，先做最小闭环：

```text
上传 Mirage demo
→ 选择己方五人
→ 解析比赛
→ 输出个人坏习惯
→ 输出团队问题
→ 输出 5 个关键回合
→ 输出 2 套专属战术
```

这个闭环能验证产品的核心价值：AI 复盘是否真的能帮玩家发现问题并给出可执行建议。
